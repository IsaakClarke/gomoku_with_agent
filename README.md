# Gomoku AI - агент в стиле AlphaZero

Игровой агент для Gomoku 15x15, обученный методом self-play глубокого обучения с подкреплением. \
Реализован с нуля на PyTorch и Pygame.

---

## С чего всё началось

Первые две попытки создать этого агента провалились.

Изначально я пытался использовать **OpenAI Gymnasium** как обёртку для игровой среды и применять к ней стандартные методы policy gradient. \
Идея казалась разумной - стандартный интерфейс, стандартные алгоритмы.

Не сработало, и вот почему:

- **Переобучение на узкий набор дебютов.** Агент быстро запоминал
  конкретные последовательности ходов и не мог обобщить знания на новые позиции.
- **Жёсткость self-play.** Gymnasium плохо подходит для реализации
  полноценного self-play с рандомизацией стартовых условий.
  Каждый эпизод начинался с одной и той же пустой доски,
  что превращало обучение в заучивание конкретных линий.
- **Отсутствие тёплого старта.** В Gomoku случайный self-play
  с нуля - это бесконечные бессмысленные позиции.
  Агент тратил огромное количество времени на мусорные партии
  и почти ничему не учился.

После двух неудачных попыток я полностью переосмыслил подход и переписал pipeline с нуля - уже без Gymnasium, ориентируясь на методологию AlphaZero, но адаптированную под ограниченные вычислительные ресурсы.

---

## Процесс обучения

### Фаза 1 - Эвристический Bootstrap

Главный вывод после первых провалов: нельзя просто бросить нейросеть в случайный self-play и ждать чуда. Ей нужна отправная точка.

Я написал **эвристического учителя** - rule-based агента, который умеет:

- находить немедленный выигрышный ход
- находить и блокировать немедленный выигрыш противника
- оценивать ходы по паттернам линий (открытые двойки, тройки, четвёрки)
- учитывать близость к центру
- распознавать угрозы структурного уровня (открытая четвёрка, двойная угроза)

Вместо жёсткого one-hot label учитель генерирует **мягкое распределение top-k ходов** с температурным сглаживанием. \
Это важно: сеть учится не только тому, *какой* ход выбрать, но и тому, *насколько* каждый ход хорош относительно остальных.

Чтобы датасет не схлопнулся в один стиль, я обучил **три варианта учителя** с разными характерами:

- **Aggressive** - высокий вес атаки, низкий вес защиты
- **Balanced** - умеренный баланс атаки и защиты
- **Exploratory** - высокая температура, больше разнообразия

Каждый вариант генерировал 300 партий с рандомизацией дебюта (первые 2 хода выбираются случайно из центральной зоны). \
Итоговый датасет - ~93 000 позиций.

**Основная сложность:** найти правильный баланс весов атаки и защиты. \
Слишком оборонительный учитель -> ~62% ничьих. \
Слишком агрессивный -> партии по 20 ходов и огромный перекос в пользу чёрных. \

Финальная конфигурация:
* Победы чёрных: 45%
* Победы белых: 25%
* Ничьи: 30%

### Фаза 2 - Предобучение на Bootstrap

На основе bootstrap-датасета была обучена **нейросеть Policy-Value Network**.

**Архитектура:**

Вход: тензор (4, 15, 15)
---> Канал 0: камни текущего игрока
---> Канал 1: камни противника
---> Канал 2: последний ход
---> Канал 3: единицы (bias-сигнал)

Stem: Conv2d(4 -> 64, kernel=3) + BatchNorm + ReLU

Backbone: \
4 Residual Block'а -> Каждый: Conv -> BN -> ReLU -> Conv -> BN + skip connection -> ReLU 

Policy Head: Conv(64 -> 2) -> BN -> ReLU -> Flatten -> Linear -> 225 логитов \
Value Head: Conv(64 -> 1) -> BN -> ReLU -> Flatten -> Linear(225 -> 64) -> ReLU -> Linear(64 -> 1) -> tanh


Доска всегда кодируется **с точки зрения текущего игрока**, а не в абсолютных терминах «чёрные/белые». \
Это ключевое решение: одна и та же сеть может играть за обе стороны без каких-либо изменений, а color bias полностью устранён на уровне представления.

Функция потерь:

L = CrossEntropy(policy_pred, policy_target)

* MSE(value_pred, итог_партии)


Обучение: AdamW, lr=1e-3, 16 эпох, batch size 64. \
Аугментация данных: 8 симметрий доски (повороты + отражения) - бесплатный ×8 буст.

После предобучения сеть уже демонстрировала разумное поведение: \
предпочтение центра на пустой доске, корректное обнаружение \
немедленных выигрышей и блоков, локально осмысленные ходы.

### Фаза 3 - Monte Carlo Tree Search

Компонент поиска реализован как **PUCT MCTS в стиле AlphaZero**:

- Случайные rollout'ы отсутствуют - листовые узлы оцениваются напрямую сетью
- Policy prior из сети направляет exploration
- Кандидатные ходы ограничены клетками в радиусе ≤ 2 от существующих камней \
  (сокращает пространство с 225 до ~20-40 кандидатов)
- Семантика value: всегда с точки зрения текущего игрока,
  знак меняется при каждом шаге backpropagation

**Ключевое открытие:** MCTS в одиночку недостаточен для форсированных тактических ситуаций. \
Даже при 250 симуляциях поиск иногда пропускает немедленные выигрыши или блоки. \
Решение - **Tactical Wrapper**, который проверяет форсированные ходы

*до* запуска MCTS:
* Можно выиграть прямо сейчас? -> играем немедленно
* Противник выиграет следующим ходом? -> блокируем немедленно
* Можно создать открытую четвёрку? -> играем немедленно
* У противника открытая четвёрка? -> блокируем немедленно
* Иначе -> запускаем MCTS


Wrapper активен в self-play, arena и финальном inference.

### Фаза 4 - Итеративный Self-Play RL

С предобученной сетью и работающим MCTS начался цикл итеративного улучшения.

**Одна итерация:**
1. Генерация self-play партий текущей лучшей моделью
2. Обучение challenger-модели на смеси данных (bootstrap + self-play)
3. Arena-оценка (challenger vs best)
4. Повышение challenger'а если winrate > 0.55

---

**Эволюция self-play генерации:**

Ранний self-play имел серьёзные проблемы:
- Все партии с пустой доски - > одинаковые дебюты каждый раз
- Чёрные выигрывали ~75% партий из-за преимущества первого хода
- Одна модель против самой себя - > коллапс стиля

Решение - **Парная генерация с фиксированным дебютом**:

Для каждой пары партий:
1. Генерируем случайный opening-префикс из 3 ходов
2. Партия A: текущая модель играет чёрными, противник белыми
3. Партия B: противник играет чёрными, текущая модель белыми
4. Обе партии стартуют **из одного и того же дебюта**

Это эффективно компенсирует преимущество первого хода и заставляет модель учиться одинаково хорошо за обе стороны.

**Пул противников (league training):**

Вместо игры только против самой себя модель играла против пула исторических чекпоинтов с весами сэмплирования:

```python
opponent_model_ids = [1, 2, 3, 4, 5, 5, 5, 0]
# Текущая лучшая встречается чаще,
# но исторические модели обеспечивают разнообразие
```

---

GPU-параллельная генерация:

Single-process self-play был слишком медленным для 1000-игровых итераций. \
Финальная архитектура генерации:

<pre>
Главный процесс:
-Task Queue (индексы пар)
-Result Queue (сэмплы)
-GPU Inference Server (батчевый инференс)
---CPU Worker 0 (MCTS + игровая логика)
---CPU Worker 1
---CPU Worker 2
---CPU Worker 3
---CPU Worker 4
</pre>

GPU Inference Server принимает запросы от всех CPU-воркеров, собирает их в батч и прогоняет через сеть за один forward pass. \
Это кардинально улучшает утилизацию GPU.

Результат: 1000 партий - с ~8 часов до ~45 минут.

Та же архитектура применена к arena-оценке: \
50 пар партий - с ~1 часа до ~10 минут.

Критическое открытие:

Train loss - ненадёжный индикатор реальной силы игры. \
Несколько раз loss почти не менялся между эпохами, но модель уверенно проходила arena. \
Обратное тоже случалось: плавные кривые потерь при нулевом прогрессе на arena. \

Arena - единственный надёжный критерий.
